=== PARAGRAPHS ===\nProject Blueprint: Nursery Resource & Progress Tracker (NRPT)\n1. Executive Summary\nA centralized web-based platform to manage 50+ simultaneous landscaping and nursery projects. The system bridges the gap between high-level executive oversight (Heat Maps/Financials) and ground-level execution (Labor/Material/Media logs). It replicates Microsoft Project’s depth with a focus on Real-Time Roll-ups, Quality Validation, and Cost Governance.\n2. User Roles & Access Control (RBAC)\nTo maintain security and data integrity, the system implements strict role-based views:\n3. Module A: Project Planning & Structure (The "MS Project" UI)\nWBS (Work Breakdown Structure): A multi-level nested grid (Project > Milestone > Sub-Milestone > Task).\nDynamic Gantt Chart: Visual timeline with dependencies (Predecessors/Successors).\nOwnership: Every task must have a Primary Owner (responsible for delivery) and can have Secondary Owners.\nWeightage System: Every task/milestone is assigned a "Weight" (%) relative to its parent. Completion of a 10% weight task contributes exactly that much to the overall project health.\nExcel Interoperability: * Bulk Upload: Import entire project plans (Tasks, Dates, Resources, BOQ) via a mapped Excel template.\nExport: Download project status or financial variance reports to Excel/PDF.\n4. Module B: Resource Engine (BOQ & BOM)\nEvery activity is linked to a Bill of Quantities (BOQ) and Bill of Materials (BOM).\nBudgeting: Architect defines "Estimated Quantity" and "Unit Price" per material.\nConsumption Tracking: Field Managers log "Actual Quantity Used."\nFinancial Masking: Field staff see that they need "500 Saplings," but never see the "₹ Price."\nAutomatic Variance: The system calculates the gap between Planned Cost vs. Actual Cost based on real-time field consumption.\n5. Module C: Field Execution & "Proof of Work"\nMultimedia Verification: A task cannot be marked "Complete" without mandatory attachments:\nPhotos: With in-app markup/annotation (drawing/circling issues).\nAudio/Video: Short clips for walk-through explanations.\nSmart Metadata: All uploads are automatically tagged with GPS Coordinates and Timestamps to verify on-site presence.\nThe Approval Loop: * Status moves from In-Progress → Pending Approval.\nSupervisor Action: If rejected, the Supervisor must provide a reason (Voice/Text).\nNotification: Instant WhatsApp/Email alert to the Field Manager for rework.\n6. Module D: Crowdsourced Defect Management\nQR Reporting: Visitors scan a QR code in a nursery zone to report issues (e.g., "Dead plant," "Broken pipe").\nHelpdesk Triage: 1. Reports land in the Helpdesk Manager's queue.\n2. Manager validates and converts the report into a "Maintenance Task."\n3. Task is assigned to a Project Manager and appears in their daily schedule.\nClosing the Loop: The reporter (if contact info provided) receives a WhatsApp update once the fix is approved.\n7. Module E: The Executive Dashboard (Heat Maps)\nFor managing 50+ projects, the dashboard uses visual "Severity" triggers.\nThe Heat Map Grid: * Green: Progress is on time and $\text{Actual Cost} \le \text{Budgeted Cost}$.\nYellow: Minor slippage in time or $<10\%$ budget overrun.\nRed: Critical delay or $>10\%$ budget overrun.\nDrill-Down Logic: * Click a Red Project → Expand to see Department (e.g., Irrigation).\nClick Department → See specific Milestone (e.g., Pump Install).\nClick Milestone → See Actual Task + Photo of the problem.\n8. Module F: Automated Notification Engine\nPre-Flight Reminders: 24 hours before any task "Start Date," the system sends a WhatsApp and Email to the Primary Owner.\nDaily Alerts: At 6:00 PM, a summary of tomorrow's work is sent to the whole team.\nEscalations: If a task is 48 hours overdue, an automated "Critical Alert" is sent to the Project Owner.\n9. Exception Handling (The "What Ifs")\nOffline Mode: If there is no internet at the nursery, the webapp (PWA) must cache photos/logs and sync automatically when back online.\nMaterial Shortage: If actual consumption exceeds 120% of the BOM, the system "locks" the task until the Architect approves a budget extension.\nOwner Absence: If a Primary Owner is on leave, the Secondary Owner automatically receives all notifications and approval rights.\nConflicting Updates: If an Excel upload happens simultaneously with a field update, the system must prompt a "Conflict Resolution" screen to the Architect.\n10. Technical Stack (Recommendation)\nFrontend: React / Next.js (for the Dashboard and PWA).\nGantt/Grid Engine: AG-Grid Enterprise or Bryntum Gantt.\nBackend: Node.js or Python (FastAPI).\nDatabase: PostgreSQL (Relational) + Redis (for real-time rollup caching).\nMessaging: Twilio API (WhatsApp) & SendGrid (Email).\nHere's the link to test the NRPT application:\n🔗 Application URL: https://fieldmap-5.preview.emergentagent.com\nDemo Credentials\nArchitect (Full Admin - Create projects, WBS, budgets):\nEmail: architect@nrpt.com\nPassword: password\nProject Owner (Executive - Heat maps, financial oversight):\nEmail: owner@nrpt.com\nPassword: password\nSupervisor/QA (Approve/reject tasks):\nEmail: supervisor@nrpt.com\nPassword: password\nField Manager (Upload work proof, submit tasks):\nEmail: field@nrpt.com\nPassword: password\nQuick Test Guide\nStart as Architect: Create a project with tasks and BOQ items\nSwitch to Field Manager: View tasks and upload media (note: costs are hidden)\nCheck Project Owner: See heat map with project health status\nTry Defect Reporting: Visit /report-defect (no login required)\nThe system has one sample project "Central Park Renovation" with a task "Install Irrigation System" already created for testing\nNRPT - Nursery Resource & Progress Tracker\nProduct Requirements Document\nOriginal Problem Statement\nA centralized web-based platform to manage 50+ simultaneous landscaping and nursery projects. Module Specifications\n## 5.1 Authentication Module\n### Features Implemented\n- JWT-based authentication with 24-hour token expiration\n- Password hashing using bcrypt\n- Role-based access control middleware\n- Login and Registration flows\n### Authentication Flow\n```\n1. User enters credentials\n2. POST /api/auth/login\n3. Server validates credentials\n4. Server generates JWT token\n5. Token stored in localStorage\n6. Token sent in Authorization header for all API calls\n7. Server validates token and extracts user role\n7. Server validates token and extracts user role\n8. Access granted/denied based on role permissions\n```\n## 5.2 Project Management Module\n### Features Implemented\n- Full CRUD operations (Create, Read, Update, Delete)\n- Executive Summary with 4 sections\n- Budget tracking with variance calculation\n- Progress percentage based on WBS completion\n- Status management (Active, Completed, On Hold)\n### Executive Summary Sections\n| Section | Purpose |\n|---------|---------|\n| What We Completed | Major milestones achieved, key deliverables |\n| Where We Stand | Current status, timeline position, resource utilization |\n| Pain Points / Risks | Blockers, delays, budget concerns |\n| Where We're Heading | Next milestones, upcoming deadlines, forecast |\n## 5.3 WBS Editor Module\n### Features Implemented\n- Tree structure with unlimited nesting levels\n- Collapse/Expand functionality for all nodes\n- Item types: Milestone, Sub-Milestone, Task\n- Weight percentage system (contribution to parent)\n- Status workflow: Pending → In Progress → Pending Approval → Completed/Rejected\n- Owner assignment (Primary)\n### Tree Structure Example\n```\n📁 LOC-01: New Nursery (Milestone, 40%)\n└── 📁 01.1 Lawn Maintenance (Sub-Milestone, 15%)\n├── ✓ Mowing, Edging, Cleaning (Task, 5%) - API Documentation\n## 7.1 Authentication Endpoints\n### POST /api/auth/register\nCreate new user account.\n**Request Body**:\n```json\n{\n"email": "user@example.com",\n"password": "securepassword",\n"name": "John Doe",\n"role": "field_manager"\n}\n```\n**Response** (200):\n```json\n{\n"access_token": "eyJhbGciOiJIUzI1NiIs...",\n"token_type": "bearer",\n"user": {\n"id": "uuid",\n"email": "user@example.com",\n"name": "John Doe",\n"role": "field_manager",\n"created_at": "2026-02-01T00:00:00Z"\n}\n}\n```\n### POST /api/auth/login\nAuthenticate existing user.\n**Request Body**:\n```json\n{\n"email": "user@example.com",\n"password": "securepassword"\n}\n```\n### GET /api/auth/me\nGet current authenticated user.\n**Headers**: `Authorization: Bearer <token>`\n---\n## 7.2 Project Endpoints\n### GET /api/projects\nList all projects.\n### POST /api/projects\nCreate new project. (Architect/Owner only)\n**Request Body**:\n```json\n{\n"name": "Project Name",\n"description": "Description",\n"budget": 1000000,\n"start_date": "2026-01-01",\n"end_date": "2026-12-31",\n"status": "active"\n}\n```\n### GET /api/projects/{project_id}\nGet single project details.\n### PUT /api/projects/{project_id}\nUpdate project. (Architect/Owner only)\n### DELETE /api/projects/{project_id}\nDelete project and all related data. (Architect only)\n---\n## 7.3 WBS Endpoints\n### GET /api/wbs/project/{project_id}\nGet WBS tree for project.\n**Response**: Nested tree structure with children arrays.\n### POST /api/wbs\nCreate WBS item.\n**Request Body**:\n```json\n{\n"name": "Task Name",\n"item_type": "task",\n"parent_id": "parent-uuid-or-null",\n"project_id": "project-uuid",\n"weight": 10,\n"status": "pending",\n"primary_owner_id": "user-uuid-or-null"\n}\n```\n### PUT /api/wbs/{item_id}\nUpdate WBS item.\n### DELETE /api/wbs/{item_id}\nDelete WBS item and all children.\n---\n## 7.4 BOQ Endpoints\n### GET /api/boq/project/{project_id}\nGet all BOQ items for project.\n**Note**: Prices masked for Field Manager role.\n### POST /api/boq\nCreate BOQ item. (Architect/Owner only)\n### PUT /api/boq/{item_id}\nUpdate BOQ item.\n**Note**: Field Manager can only update `actual_quantity`.\n### DELETE /api/boq/{item_id}\nDelete BOQ item. (Architect/Owner only)\n---\n## 7.5 Defect Endpoints\n### GET /api/defects/project/{project_id}\nGet all defects for project.\n### POST /api/defects\nCreate defect. (Public endpoint for QR reporting)\n### PUT /api/defects/{defect_id}\nUpdate defect.\n### DELETE /api/defects/{defect_id}\nDelete defect. (Architect/Owner/Supervisor only)\n---\n## 7.6 Approval Endpoints\n### GET /api/approvals/pending\nGet all items pending approval.\n### POST /api/approvals/{item_id}\nProcess approval action.\n**Request Body**:\n```json\n{\n"action": "approve",  // or "reject"\n"reason": "Rejection reason"  // required if rejecting\n}\n```\n---\n## 7.7 Dashboard Endpoints\n### GET /api/dashboard/stats\nGet dashboard statistics.\n**Response**:\n```json\n{\n"total_projects": 3,\n"total_budget": 35500000,\n"total_actual_cost": 10500000,\n"total_variance": 25000000,\n"projects": [\n{\n"id": "uuid",\n"name": "Project Name",\n"budget": 10000000,\n"actual_cost": 2500000,\n"progress": 25.0,\n"health": "green",\n"variance": 7500000\n}\n]\n}\n```\n---\n# 8.
